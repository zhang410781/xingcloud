# 告警精准研判

> 状态：当前已实现
> 架构：单服务内的逻辑阶段，不是独立 Agent 服务

## 1. 触发规则

自动精准研判只针对 Pod/容器类 Warning、Critical 告警：

1. 告警首次触发后立即发送原始告警摘要。
2. 告警持续活跃两分钟后进入一次精准研判。
3. 两分钟内恢复则取消深度研判。
4. 研判完成后发送独立更新；恢复通知继续走原流程。

同一告警指纹只保留一个等待或运行中的研判任务。手动研判不受自动范围限制。

## 2. 逻辑阶段

状态依次为：

`waiting_persistence → collecting_metrics → collecting_k8s → collecting_logs → synthesizing → completed/partial/cancelled`

这些阶段对应 Agent‑1/2/3/4 的职责拆分，但全部运行在现有 Django 后端和 Scheduler 中。

## 3. 指标证据

指标窗口为 `starts_at - 10 分钟` 到研判执行时刻。查询按业务上下文、namespace、Pod、container 和 node 尽量精确过滤。

Pod/容器研判重点采集：

- 告警规则原始指标。
- 窗口内重启增量。
- CPU 与内存峰值。
- CPU Request、内存 Limit。
- OOMKilled 指标。
- 节点磁盘 I/O 和内存可用率。

生命周期累计重启只作为背景，不能代替时间窗口内的重启增量。

## 4. K8S 深度证据

平台通过 Kubernetes API 读取目标 Pod 的 describe 等价信息：

- phase、Ready、waiting/terminated reason、exit code 和重启次数。
- 镜像、Request、Limit 和所在节点。
- Warning Event。
- 当前日志和 previous 崩溃日志。

单项读取失败时记录原因并继续其他阶段，不依赖本地 `kubectl`。

## 5. 日志与变更

日志主窗口从告警 `starts_at` 到执行时刻，并按 namespace、Pod、容器、服务等维度过滤。关键证据只保留 Error、Warning、异常堆栈、OOM、GC、连接、存储和启动失败等模式；普通成功访问日志不作为关键根因证据。

平台同时查询时间窗口内的发布、任务和变更记录，作为候选相关证据。

## 6. 根因与置信度

确定性模式优先识别 OOMKilled、探针失败、CrashLoop、镜像/配置错误、调度失败、PVC、节点和网络异常。

- 单一独立来源最高 49%。
- 两类独立来源最高 79%。
- 80% 以上需要至少三类独立来源和直接异常证据。
- 模型只能整理已有证据，并受平台计算的置信度上限约束。

无法形成证据链时固定说明“告警仍活跃但无法确诊”或“已恢复但无法确诊”，不得凭经验补充无证据原因。

## 7. 输出

网页展示告警概览、影响范围、指标、K8S 状态、日志与事件、候选根因、反向证据、置信度和 P0/P1/P2 建议。飞书等渠道发送同结构的 Markdown 摘要，不展示模型思维链、完整 PromQL 或无限量原始日志。

## 8. 当前边界

- 不保存独立 K8S 现场快照。
- 不等待告警未来二十分钟的数据。
- 不自动执行修复，只能生成需人工确认的任务草稿。
