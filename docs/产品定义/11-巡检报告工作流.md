# 巡检报告工作流

## 1. 产品定位

巡检报告是通知资源的一部分，用于将平台确定性巡检结果按计划发送给运维人员。它不是告警规则，不创建虚假告警，也不依赖模型主动调用工具。

## 2. 配置入口

入口：`可观测 → 告警规则 → 通知资源 → 巡检报告`。

一个巡检报告计划固定配置：

- 计划名称和业务上下文。
- 每天或每周发送；每周可选择星期一至星期日。
- 发送时间，默认使用 Asia/Shanghai 时区。
- 巡检范围只保留两类：
  - 集群综合巡检：侧重 K8S 集群、控制平面、节点、Pod、工作负载、存储和集群日志。
  - 服务器巡检：侧重物理机或虚拟机的 CPU、内存、负载、磁盘、inode、网络、TCP 和磁盘 I/O。
- 证据时间窗。
- 一个或多个通知渠道。
- 一个或多个接收人、接收组。

接收人通过姓名、接收渠道和必要联系方式登记：邮件需要邮箱，短信或语音需要手机号；飞书、钉钉和企微复用通知资源中配置的机器人渠道，不填写个人用户 ID。选择接收人时只显示姓名和渠道，不展示内部主键。

## 3. 执行流程

```text
调度器发现到期计划
  → 根据业务上下文和巡检范围解析 Prometheus、K8S、日志源和资产范围
  → 运行 collect_observability_evidence
  → 生成健康分、集群概览、异常项、建议和缺失证据
  → 渲染巡检报告 Markdown
  → 通过邮件、飞书、钉钉、企微、短信或语音渠道发送
  → 保存执行状态、报告内容和各渠道发送结果
  → 计算下一次发送时间
```

模型不可用时仍发送确定性证据报告。任何单项证据源失败时，报告标记为部分完成并明确显示未获取证据，不生成虚假结论。

## 4. 状态与安全

- 计划状态：启用、停用。
- 最近执行状态：未执行、成功、部分成功、失败。
- 支持“立即发送”用于上线前验证。
- 启用计划必须至少选择一个通知渠道，以及一个接收人或接收组。
- 接收人或接收组被计划引用时不可直接删除。
- 飞书渠道继续强制签名校验。
- 巡检报告只读，不执行自动修复；处理建议只能进入人工确认流程。

## 5. 报告内容

- 业务上下文、巡检范围和生成时间。
- 总体结论与健康分。
- 集群综合巡检展示节点 Ready 情况、Pod 状态分布和 K8S 资源证据。
- 服务器巡检展示主机数量、CPU、内存、负载、磁盘、网络、TCP 和 I/O 证据，不混入 Pod 或控制面内容。
- 严重、警告和提示发现项。
- 指标、K8S、日志、资产、拓扑和变更证据。
- 建议操作和未获取证据原因。
- 平台下钻提示。

健康分沿用固定巡检规则：严重问题扣20分、警告扣8分、提示扣2分；同一对象的同一问题类型只扣一次，日志样本数量不直接叠加扣分。
