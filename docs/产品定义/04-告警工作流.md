# 告警管理 — 产品工作流

## 1. 产品目标

告警系统是 SRE 数字员工的"神经系统"。它负责捕获异常信号、降噪去重、自动关联多源信息研判根因、并按策略将处理过的告警分发给对应的人。

告警智能体是 7 个智能体的核心入口——告警触发后自动唤醒故障研判智能体，联动日志分析智能体，最终输出处理建议。异常检测智能体作为前置预检，在指标偏离基线时提前生成低级别告警。Runbook 智能体作为后置输出，在根因定位后自动推荐处理步骤。

## 2. 告警数据模型

### 2.1 告警规则（AlertRule）

规则定义"什么条件下触发告警"：

| 字段 | 说明 |
|------|------|
| 数据源 | 关联的 Prometheus/ClickHouse 数据源 |
| 查询语句 | PromQL/日志关键字/SLA 条件 |
| 条件 | 比较器（> / < / >= / <=）+ 阈值 + 持续时间 |
| 严重级别 | critical / warning / info |
| 分组标签 | 用于静默匹配、通知路由、聚合的标签集 |
| 评估间隔 | 每隔多少秒评估一次 |
| 静默调度 | 可配置的静默时间窗口（重复：每天/每周） |
| 升级分钟 | 持续未恢复多少分钟后升级通知 |
| 通知配置 | 关联的通知策略 |

### 2.2 告警事件（Alert）

规则触发后产生的实例：

| 字段 | 说明 |
|------|------|
| 状态 | firing / resolved / acknowledged |
| 标签 | 继承自规则的标签 + 动态注入的实例标签 |
| 触发时间 | 规则条件首次满足的时间 |
| 恢复时间 | 规则条件不再满足的时间 |
| AI 分析状态 | pending / analyzing / completed |
| 确认人 | 谁确认了这个告警 |

### 2.3 告警静默（AlertSilence）

替代旧版的 MuteRule + InhibitionRule：

| 字段 | 说明 |
|------|------|
| 名称 | 可读标识 |
| 匹配器 | 标签键值对匹配（支持正则） |
| 时间窗口 | 开始时间 + 结束时间（支持重复：每天/每周/一次性） |
| 原因 | 说明为什么静默 |

## 3. 完整告警流转链路

```
┌─────────────────────────────────────────────────────────────┐
│                    1. 规则定义阶段                              │
│  用户通过向导创建告警规则                                      │
│    步骤1：选择监控对象（MySQL/Redis/K8S/主机/日志/SLA）       │
│    步骤2：选择规则模板（预设模板列表，展示表达式和默认阈值）    │
│    步骤3：配置触发条件（阈值/持续时长/级别/分组标签）          │
│    步骤4：配置响应行为（通知策略/AIOps 分析/静默/升级）       │
│    步骤5：试运行预览（执行一次查询看当前值） + 保存           │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                    2. 规则评估阶段                              │
│  后台调度器按评估间隔周期执行                                  │
│     → 向数据源发起查询                                        │
│     → 将结果与阈值比较                                        │
│     → 不满足条件 → 跳过                                       │
│     → 满足条件 → 创建/更新告警事件                            │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                    3. 告警智能体（降噪阶段）                    │
│    告警事件进入降噪管线                                       │
│     步骤1：去重 —— 相同规则+相同标签的已有 firing 告警不重复   │
│     步骤2：风暴检测 —— 短时间大量同类告警自动聚合为"风暴"     │
│     步骤3：静默匹配 —— 匹配 AlertSilence 的告警标记为静默     │
│     步骤4：分级路由 —— 按级别+标签决定走哪条通知策略          │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                    4. 故障研判（确定性证据阶段）                │
│    服务端按业务上下文直接执行，不依赖模型主动调用工具：        │
│     步骤1：告警指标 —— 当前值、阈值、趋势和历史异常           │
│     步骤2：K8S 证据 —— 对象状态、Warning Event、重启前日志    │
│     步骤3：日志证据 —— 告警前5分钟至后2分钟，基线为前30~5分钟 │
│     步骤4：变更与拓扑 —— 最近发布、事件和上下游依赖           │
│     步骤5：根因研判 —— 基于结构化证据生成候选原因和置信度上限 │
│     输出：事实、推断、缺失证据和待确认建议写入告警分析         │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                    5. 通知分发阶段                              │
│  按通知策略将告警+AI 分析发送到对应渠道：                      │
│    企业微信机器人 / 飞书 / 邮件 / 自定义 Webhook               │
│   通知按生命周期分为触发、研判、升级和恢复：                   │
│    - 告警摘要：级别、业务上下文、集群、对象、当前值与阈值      │
│    - 研判结论：事实与候选原因分离，展示证据覆盖和置信度         │
│    - 建议操作：最多3项，高风险操作只生成待确认任务草稿          │
│    - 恢复通知：恢复时间、持续时间、恢复证据和残余风险           │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                    6. 人工处置阶段                              │
│  值班 SRE 收到通知 → 查看详情：                                │
│    确认/解决/创建工单/创建静默                                 │
│  如果配置了升级策略且超时无人确认：                             │
│    自动升级通知给更高层级（架构师/主管）                       │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                    7. 恢复与复盘                                │
│  规则条件不再满足 → 告警自动 resolved                          │
│  告警事件进入历史库 → 供告警回顾、SLA 计算、趋势分析          │
└─────────────────────────────────────────────────────────────┘
```

## 4. 通知策略

### 4.1 通知渠道

| 渠道 | 状态 | 配置方式 |
|------|------|---------|
| 企业微信机器人 | 🟢 可用 | Webhook URL |
| 飞书 | 🟢 可用 | Webhook URL |
| 邮件 | 🟢 可用 | SMTP 配置 |
| 自定义 Webhook | 🟢 可用 | 任意 HTTP endpoint + 自定义模板 |

### 4.2 通知路由规则

```
告警事件（级别+标签）
   ↓
匹配通知策略（按严重级别 / 标签 / 时间窗口）
   ↓
分发到指定接收组（AlertRecipientGroup）
   ↓
组内成员通过各自渠道接收
```

## 5. 模块间依赖

| 输入来源 | 输出去向 |
|----------|----------|
| 可观测性数据源 | 告警规则定义（查询+条件） |
| CMDB 资产 + 业务上下文 | 规则向导中监控对象的范围选择 |
| 异常检测智能体 | 前置预检结果 → 低级别告警事件 |
| 告警引擎评估结果 | 故障研判智能体（触发 AI 分析） |
| 告警事件 | 通知分发 → 企微/飞书/邮件 |
| 告警事件 | Runbook 智能体（匹配处理步骤） |
| 告警历史 | 运行概览（SLA 计算） |

## 6. 常见场景

| 场景 | 操作路径 |
|------|---------|
| MySQL 连接数突增 | 告警规则触发 → AI 研判看到慢查询增加 → 通知 DBA |
| K8S Pod 频繁重启 | 日志分析智能体检索 CrashLoopBackOff → 故障研判定位 OOM → 建议调大内存限制 |
| 磁盘使用率超 90% | 升级策略生效 → 运维未确认 → 自动升级通知主管 |
| 某业务无告警但用户报障 | 手动发起 AIOps → "分析 payment 服务最近异常" → 故障研判智能体联动执行 |
| 大促前批量添加告警规则 | 选择模板 → 调整阈值（调低响应灵敏度）→ 批量导入 → 试运行验证 |

## 7. 边界与限制

- 当前不支持告警自愈（告警→自动执行脚本→恢复），需人工确认后操作
- 不支持外部告警写入（不能通过 Webhook 接收其他系统的告警）
- 风暴检测当前为规则级聚合，不跨规则
- 日志只能作为一种证据源：仅日志证据时研判置信度最高49%，两类独立证据最高79%，80%以上至少需要三类独立证据和直接异常事实
- 查询不到日志不代表系统正常，必须在分析和通知中展示“日志证据缺失”或“当前查询未命中”
- 通知排版由后端固定模板控制，Skill只定义研判流程和结构化输出，不直接生成渠道最终格式
