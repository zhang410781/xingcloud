# 巡检报告

> 状态：当前已实现

## 1. 巡检类型

### 集群综合巡检

以 K8S 集群为中心，同时覆盖节点服务器，检查：

- 节点数量、Ready、CPU、内存、负载、磁盘、inode、网络和 I/O。
- Pod 状态、重启排行和异常容器。
- Deployment、StatefulSet、DaemonSet 与 PVC。
- API Server、Requests/Limits 和资源余量。
- Warning Event、异常日志和近期变更。

### 服务器巡检

以物理机或虚拟机为中心，重点检查 CPU、内存、负载、磁盘/inode、网络、I/O、进程和异常日志，不要求存在 K8S 范围。

## 2. 证据来源

巡检通过确定性管线直接读取业务上下文绑定的 Prometheus、Kubernetes API、日志源和资产，不依赖模型主动调用工具。单个来源失败时仍生成部分报告并列明缺失原因。

## 3. 报告内容

- 巡检范围与时间窗。
- 健康分和结论。
- 节点状态表。
- Pod 状态与重启排行。
- 工作负载与存储统计。
- 指标采样表。
- 日志与事件。
- 严重、警告和提示发现项。
- 建议操作和与上次报告的变化。

健康分按问题指纹去重扣分：严重问题 20 分、警告 8 分、提示 2 分。

## 4. 计划与发送

巡检报告计划位于告警规则的通知资源中，属于全局通知资源，可跨业务上下文查看。创建计划时选择：

- 业务上下文。
- 集群综合巡检或服务器巡检。
- 每日或每周周期、时间和时区。
- 通知渠道、接收人或接收组。

“立即发送”会实时执行确定性采集、生成报告并发送，不是只发送缓存模板；在数据源响应较快时可能数秒内完成。

## 5. 输出格式

- 平台保存结构化 JSON、报告块和 Markdown。
- 飞书/微信等渠道发送受长度控制的完整 Markdown 报告。
- 当前不生成 PDF。

## 6. 当前边界

- 巡检报告只读，不自动修复。
- 日报/周报的价值是展示新增或恶化问题；无证据时不推断健康。
