# AIOps 智能体引擎 — 产品工作流

## 1. 产品目标

AIOps 是 SRE 数字员工的"大脑"。它通过 Action 明确任务目标，通过 Skill 约束取证和输出，通过 MCP 或平台内置工具访问真实数据。固定巡检和自动告警研判由服务端确定性执行证据管线，不依赖模型主动调用工具；LLM 只负责在证据边界内进行归纳和解释。

7 个智能体（巡检、故障研判、日志分析、告警、Runbook、异常检测、变更）的智能能力都由 AIOps 引擎提供或触发。

## 2. 7 大智能体的工作模式

### 2.1 巡检智能体

**触发方式**：用户即时巡检 / 告警研判联动；定时周报不在当前一期范围

**工作流**：
```
用户选择业务上下文或告警触发
   → Action 路由到 k8s.inspect
   → 根据问题范围选择巡检 Profile：
       cluster / node / workload / service / control_plane
   → 服务端解析业务上下文绑定的唯一 Prometheus、日志源和 K8S 集群
   → 确定性并行采集：
       指标趋势 / K8S 状态 / Warning Event / 日志 / 变更 / 资产拓扑
   → 代码计算异常项、去重指纹和健康评分
   → Skill 约束证据解释和建议结构
   → 输出：健康结论、集群概览、严重/警告/提示项、关键证据、缺失证据和下钻入口
```

**输出示例**：
```
## 🟢 数智平台 K8S 集群巡检 — 健康评分 92/100

### 当前异常项（2项）
| 异常 | 级别 | 关键证据 | 建议 |
|------|------|---------|------|
| Node disk-pressure | 警告 | 节点磁盘可用空间持续低于阈值 | 先确认日志与镜像占用，再生成待确认清理任务 |
| Pod 重启增加 | 警告 | 5分钟内重启3次，出现OOM日志模式 | 核对Limits、退出原因和最近发布 |

### 缺失证据
- 未获取到目标Pod的上次崩溃日志，当前根因仍需核实。
```

### 2.2 故障研判智能体

**触发方式**：告警触发 / 用户手动提问

**工作流**：
```
告警触发（或用户提问）
   → 解析上下文：获取告警规则、标签、触发时间、关联资产
   → 确定数据源范围：通过知识图谱获取业务上下文绑定的指标源+日志源
   → 第一阶段查询（预算：8 条 PromQL + 2 条日志查询）：
       指标证据包 → 告警原始指标 + RED 指标 + K8S 运行态
       日志分析 → 主窗口为告警前5分钟至后2分钟，基线窗口为告警前30分钟至前5分钟
       K8S 事件 → 查询 Pod/Node 近期事件
   → 第二阶段扩展（仅在证据指向明确时触发）：
       上下游依赖指标 → 入口/出口/网关
       变更记录 → 近期发布/工单
   → LLM 根因分析：
       综合多源证据 → 得出根因假设 + 置信度评分
   → 输出分析结论：
       写入告警事件 AI 分析字段
       随通知消息一并发送
```

**输出示例**：
```
## 🤖 AI 根因分析

**根因**：payment-order 服务 Pod OOMKilled（置信度：85%）
**证据**：
- [异常] 容器内存使用率 98%（基线 45%）
- [异常] Pod 重启 3 次/5min
- [正常] 上游 Nginx 请求量平稳
- [正常] MySQL 响应时间正常

**建议**：调整 Pod memory limit 从 512Mi 到 1Gi
```

### 2.3 日志分析智能体

**触发方式**：被故障研判智能体调用 / 用户直接提问

**工作流**：
```
收到分析请求（关键词/时间窗/范围）
   → 通过知识图谱确定目标日志源
   → 生成查询参数（时间范围 + 关键词 + 过滤标签）
   → 执行日志查询（Loki/ELK/CH）
   → 结果后处理：
       提取异常模式（ERROR/Exception/OOM 等出现频率）
       生成时间线（按时间分布展示关键日志行）
       去重/聚合（同类日志合并展示）
   → 返回结构化日志证据包
```

**输出示例**：
```
## 📋 日志证据包

**匹配**：搜索 "OOMKilled" + "payment-order" 在 2026-07-17 14:00-14:30

| 时间 | 级别 | 来源 | 内容 |
|------|------|------|------|
| 14:02:15 | ERROR | payment-order-7d8f9 | resource limit exceeded: memory 512Mi |
| 14:05:22 | WARN | payment-order-7d8f9 | OOMKilled |
| 14:05:23 | INFO | kubelet | killing container payment-order |

**趋势**：重启次数 0→3 跳变（14:00-14:05 集中发生）
```

### 2.4 告警智能体

**触发方式**：告警引擎事件驱动

**工作流**（降噪管线）：
```
收到原始告警事件
   步骤1：去重 —— 相同 fingerprint 的已有告警不重复创建
   步骤2：风暴检测 —— 同规则 5min 内 >10 次触发 → 合并为"xxx 风暴"
   步骤3：静默匹配 —— 比对 AlertSilence 匹配器 → 命中则标记静默
   步骤4：通知路由 —— 按规则配置的策略分发给对应接收组
   步骤5：触发故障研判 —— 如规则配置了 AI 分析 → 异步启动故障研判智能体
```

### 2.5 Runbook 智能体

**触发方式**：故障研判触发匹配 / 用户主动搜索

**工作流**：
```
故障研判输出根因
   → Runbook 引擎以告警标签 + 根因摘要为输入
   → 在 Runbook 库中检索匹配
   → 返回 Top-3 推荐列表（带置信度排序）
   → 随通知消息发送给 SRE
   → SRE 查看详情 → 阅读步骤 → 关联任务脚本可一键执行
```

### 2.6 异常检测智能体

**触发方式**：定时扫描（默认每15min）/ 告警规则智能模式触发

**工作流**：
```
拉取目标指标最近 24h 历史数据
   → 执行多算法检测（3-Sigma / EWMA / IQR / Isolation Forest）
   → 投票机制（≥2 算法通过判定异常）
   → 正常 → 跳过
   → 异常 → 生成异常记录（评分+方向+算法明细）
            → 可选：创建 info/warning 告警事件
            → 可选：触发故障研判简要分析
```

## 4. 安全与审计

| 约束 | 说明 |
|------|------|
| 只读操作 | 查询类直接返回结果 |
| 写操作 | 必须生成草稿并等待用户确认（Action Handler） |
| 审计追踪 | 每条 LLM 调用、每个 MCP 工具执行、每个 Action 确认/取消记录入库 |
| 证据约束 | LLM 只能基于工具返回的结构化事实做结论，不能编造 |
| 权限继承 | MCP 工具继承用户的 RBAC 权限，跨权限调用自动拒绝 |

## 4.1 Action、Skill 与 MCP 的边界

```
Action：定义当前要完成什么任务，并限定风险、上下文和允许工具
Skill：定义取证顺序、判断口径、巡检 Profile 和输出契约
MCP：提供告警、指标、日志、K8S、知识图谱、事件和任务等真实能力
后端代码：执行固定巡检、异常检测、置信度上限、通知渲染和安全审计
```

实际可用工具为“启用 MCP、Action 允许工具、MCP 白名单、用户 RBAC、功能开关与安全策略”的交集。Skill只能声明依赖，不能绕过 Action 或 MCP 获得额外权限。

## 5. 模块间依赖

| 输入来源 | 输出去向 |
|----------|----------|
| CMDB（资产范围） | MCP 工具的数据来源 |
| 可观测性（指标/日志） | MCP 工具的数据来源 |
| 告警引擎（告警事件） | 故障研判触发入口 |
| 异常检测算法结果 | 告警引擎（异常事件） |
| 知识图谱（业务上下文） | 数据源范围约束、资产关联 |
| Runbook 引擎（剧本库） | 任务中心（执行剧本步骤） |
| 任务中心（执行记录） | 巡检报告的执行依据 |
