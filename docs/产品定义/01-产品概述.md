# SRE 数字员工 — 产品定义

> 版本：v1.0 | 状态：内部工具 | 最后更新：2026-07-17

---

## 1. 产品一句话定位

**Xing-Cloud 是一个面向生产环境的 SRE 数字员工平台，通过 7 个 AI 智能体（巡检、故障研判、日志分析、告警、Runbook、异常检测、变更）将可观测性数据转化为可执行的运维结论和动作。**

它不是传统的监控告警 UI，也不是 ChatOps 机器人，而是"有自主工作能力的运维成员"——能够按周期执行巡检、收到告警后自动关联多源信息定位根因、在沟通群中给出分析结论和处理建议。

---

## 2. 目标用户

| 角色 | 职责 | 使用方式 |
|------|------|----------|
| **SRE / 运维** | 日常值班、故障应急、容量规划 | Web 工作台 + 群聊机器人推送 |
| **研发** | 排查自身服务问题、查看日志和调用链 | Web 工作台 |
| **运维管理者** | 查看整体健康度、SLA、审计记录 | 运行概览驾驶舱 + 定期报告 |
| **平台管理员** | 配置数据源、CMDB 资产、RBAC 权限 | Web 后台 |

## 3. 7 大智能体定义

这是平台的核心能力，所有模块围绕它们组织。

### 3.1 巡检智能体

| 属性 | 定义 |
|------|------|
| **用途** | 按计划（周/日）自动巡检集群、中间件、核心服务的健康状态，输出巡检报告并推送到指定群/邮箱 |
| **触发方式** | 定时调度（Cron）或手动触发 |
| **所需数据** | CMDB 资产列表、K8S 集群状态、指标数据（PromQL）、日志聚合 |
| **输出** | 结构化巡检报告（PDF/Markdown格式）：健康评分、异常项列表、趋势对比、变更记录 |
| **依赖模块** | CMDB（知道巡检对象） → 可观测性（获取健康指标） → 任务中心（执行巡检脚本）→ 通知（推送报告） |

### 3.2 故障研判智能体

| 属性 | 定义 |
|------|------|
| **用途** | 告警触发后，自动关联指标、日志、K8S 事件、变更记录等多源信息，定位根因并给出置信度评分 |
| **触发方式** | 告警事件触发 / 用户手动提问 |
| **所需数据** | 全域：指标证据包、日志片段、K8S Pod 事件、CMDB 资产关联关系、业务上下文 |
| **输出** | 根因分析报告：异常指标列表（带趋势）、相关日志、影响范围、置信度评分、修复建议 |
| **依赖模块** | 告警（触发入口）→ AIOps（LLM 分析 + MCP 工具）→ 可观测性（指标/日志证据）→ CMDB（资产关联）→ 知识图谱（业务拓扑） |

### 3.3 日志分析智能体

| 用途 | 定义 |
|------|------|
| **用途** | 自动从告警或用户提问中提取关键信息，搜索关联的日志数据，返回结构化日志证据 |
| **触发方式** | 被故障研判智能体调用 / 用户直接在 AIOps 中提问 |
| **所需数据** | 日志数据源（Loki / ELK / ClickHouse）、容器元数据（Pod 名、命名空间） |
| **输出** | 日志证据包：匹配条目数、关键词提取、异常趋势、关联时间线 |
| **依赖模块** | 可观测性-日志源（数据来源）→ AIOps（LLM 理解自然语言 → 生成查询参数） |

### 3.4 告警智能体

| 属性 | 定义 |
|------|------|
| **用途** | 接收原始告警事件，执行去重/聚合、风暴检测、分级路由，匹配通知策略后发送到对应渠道 |
| **触发方式** | 告警引擎评估规则触发 |
| **所需数据** | 告警规则定义、静默策略、通知渠道配置、分级路由规则（严重级别+标签） |
| **输出** | 已降噪的告警事件 + 通知消息（企微/飞书/邮件/Webhook） |
| **依赖模块** | 告警规则（触发）→ 告警引擎（评估）→ 降噪管线（去重/聚合/静默）→ 通知分发 |

### 3.5 Runbook 智能体

| 属性 | 定义 |
|------|------|
| **用途** | 故障研判定位根因后，自动匹配运维剧本（SOP），推荐给 SRE 确认执行；SRE 也可主动搜索 Runbook 获取故障处理步骤 |
| **触发方式** | 被故障研判智能体调用 / 用户主动搜索 / 手动创建 |
| **所需数据** | 故障上下文（告警标签、根因总结）、Runbook 库（版本化剧本、关联的 task、关联的 session） |
| **输出** | 匹配的 Runbook 推荐列表（按置信度排序）；已发布的 Runbook 可直接在任务中心执行 |
| **依赖模块** | 故障研判（触发入口）→ Runbook 引擎（匹配+推荐）→ 任务中心（执行剧本步骤）→ 审计（全链路记录） |

### 3.6 异常检测智能体

| 属性 | 定义 |
|------|------|
| **用途** | 对指标历史数据执行多算法异常检测（3-Sigma、EWMA、IQR、Isolation Forest），在固定阈值告警触发前提前发现指标偏离基线的趋势，作为告警智能体的前置补充 |
| **触发方式** | 定时评估 / 被告警规则中的"智能检测"条件触发 |
| **所需数据** | 指标历史数据（PromQL 查询结果），无需配置阈值 |
| **输出** | 异常评分 + 异常方向（突增/突降/波动） + 算法投票结果 |
| **依赖模块** | 可观测性-指标源（数据来源）→ 异常检测算法 → 告警引擎（生成异常事件）→ 故障研判（可选分析） |

### 3.7 变更智能体

| 属性 | 定义 |
|------|------|
| **用途** | 在每次应用发布或变更操作前，自动评估变更风险；发布中实时监控指标异常；发布后对比前后基线，检测回退信号 |
| **触发方式** | 工单审批通过后自动触发 / 用户手动发起变更评估 |
| **所需数据** | 变更范围（服务/配置/集群）、CMDB 关联资产、历史告警变更记录、发布前后实时指标 |
| **输出** | 变更前风险评估报告、变更中异常信号、变更后指标对比报告 |
| **依赖模块** | 工单系统（触发入口）→ CMDB（影响范围推导）→ 可观测性（指标对比）→ 告警引擎（异常检测）→ 通知（报告推送） |

## 5. 核心设计原则

1. **AI 是工作伙伴，不是搜索引擎。** 每个智能体都必须基于平台内可控工具（MCP）返回的结构化事实做结论，不能自由编造。
2. **可审计是前提。** 每一次智能体调用、每一个 Action 执行、每一份报告都要可追溯。
3. **确认制降噪。** 只读查询直接返回，写操作（执行、变更、发布）必须生成草稿+用户确认。
4. **生产环境优先。** 不区分 dev/staging/prod，统一按生产资产管理。新装不预置任何数据。

## 6. 模块能力矩阵

| 模块 | 服务对象 | 关键数据 | 下游 |
|------|----------|----------|------|
| CMDB | 所有智能体 | 资产、拓扑、关联关系 | 知识图谱、任务中心 |
| 可观测性 | 巡检/故障研判/日志分析/异常检测 | 指标、日志、看板 | 告警引擎、AIOps |
| 告警引擎 | 告警智能体 | 告警规则、静默策略 | 通知、故障研判 |
| 知识图谱 | 故障研判 | 业务上下文、资产关联 | AIOps 分析范围约束 |
| AIOps | 故障研判/日志分析/巡检/Runbook/变更 | MCP 工具、LLM、Skill | 用户（分析结果） |
| 异常检测算法 | 异常检测智能体 | 指标历史数据 | 告警引擎（异常事件） |
| Runbook 引擎 | Runbook 智能体 | 剧本、版本、源任务/Session | 任务中心（执行步骤）|
| 变更风险评估 | 变更智能体 | 变更范围、CMDB 资产、历史告警 | 可观测性（指标对比）|
| 任务中心 | 巡检/Runbook | 主机、K8S、脚本、调度 | 执行记录 → 审计 |
| 工单系统 | 故障研判 | 审批流、发布工单 | 变更记录 → 审计 |
| RBAC+审计 | 所有模块 | 用户、角色、权限 | 审计日志 |
