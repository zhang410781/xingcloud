# 产品概述

> 状态：当前产品定义
> 产品名称：Xing-Cloud 运维智能体平台

## 1. 产品定位

Xing-Cloud 面向生产运维场景，把 CMDB 资产、Prometheus 指标、日志、Kubernetes、告警、任务和工单统一到同一平台，并通过确定性证据采集和可配置模型辅助运维人员完成查询、巡检、告警研判和受控处置。

“数智平台”“智能运维平台”等名称是部署实例或业务上下文名称，不是另一套产品架构。

## 2. 目标用户

| 用户 | 主要工作 |
| --- | --- |
| 运维/SRE | 登记资产、接入数据源、配置告警、巡检和处置故障 |
| 值班人员 | 查看告警、接收通知、认领事件并参考研判证据 |
| 研发人员 | 在授权范围内查看服务指标、日志和 K8S 状态 |
| 管理人员 | 查看运行概览、风险、任务、工单和审计记录 |
| 平台管理员 | 管理用户权限、业务上下文、模型、Skill、MCP 和通知资源 |

## 3. 产品模块

| 一级入口 | 当前功能 |
| --- | --- |
| 运行概览 | 平台整体态势、资产、告警、任务、工单和风险摘要 |
| AIOps | 智能助手、知识图谱、智能体配置、智能体审计 |
| 可观测性 | 平台总览、告警中心、告警规则、监控看板、指标查询、日志查询、数据源 |
| 任务中心 | 任务工作台；计划任务受功能开关控制 |
| 资产管理 | 资产登记、中间件资产 |
| 工单系统 | 应用发布、事务工单、审批流 |
| 平台管理 | K8S 集群 |
| 系统管理 | 用户管理、操作审计、模块管理 |

## 4. 核心设计

1. **先登记，后组装。** 先在 CMDB、数据源和 K8S 管理中登记资源，再创建业务上下文。
2. **业务上下文统一范围。** 监控、日志、告警、巡检、资产和助手共享同一业务上下文。
3. **证据优先。** 巡检与告警研判先采集指标、K8S、日志、事件、资产和变更证据，再由模型整理。
4. **受控执行。** 写操作必须经过权限、风险检查和人工确认，并进入任务或工单流程。
5. **新环境为空。** 平台不预置虚假资产；内置规则和看板不代表已有真实监控数据。

## 5. 技术形态

- 前端：Vue 3、Vue Router、Pinia、Element Plus。
- 后端：Django 与 Django REST Framework 单体应用。
- 核心模块：`ops`、`aiops`、`cmdb`、`rbac`。
- 调度：统一 Scheduler 负责数据源健康检查、告警规则、研判任务和巡检计划。
- 数据接入：Prometheus、Loki、Elasticsearch、ClickHouse 和 Kubernetes API。

## 6. 产品边界

### 当前已实现

- CMDB、业务上下文、固定观测看板、指标/日志查询。
- 告警规则、通知策略、接收资源、告警精准研判。
- 集群综合巡检、服务器巡检及日报/周报发送。
- AIOps 助手、Action、Skill、MCP、模型供应商和 Runbook 管理。
- K8S 管理、任务、应用发布、事务工单、RBAC 与审计。

### 有限支持或内部能力

- 看板定义具备后端 CRUD、导入和导出接口，但当前固定看板页面不开放编辑器。
- Docker 主机和容器相关代码仍在仓库中，但不属于当前菜单和交付功能。
- 异常检测算法用于告警与巡检证据，不是独立产品入口。

### 当前未实现

- APM、Trace、服务调用链。
- 独立部署的多 Agent 微服务。
- 无人值守自动修复或自动回滚。
- PDF 巡检报告和 BI 式拖拽看板。
