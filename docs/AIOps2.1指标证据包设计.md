# AIOps 2.1 指标证据包设计

## 背景

现有平台已经具备指标数据源和 PromQL 查询能力，AIOps 在告警根因分析中需要使用指标证据，但不能把“生成 PromQL”和“判断根因”都交给模型自由发挥。自由生成容易出现查询过多、查询过少、跨环境误查、高基数爆炸、时序明细过大、结论不可审计等问题。

AIOps 2.1 的目标是把指标查询沉淀为平台后端可控的证据链：后端负责范围约束、模板选择、预算控制、查询执行、结果压缩和权限校验；模型只负责解释结构化证据是否支持某个根因假设。

## 设计原则

1. 后端不生成无限自由 PromQL，只生成受预算约束的指标查询计划。
2. 优先查询告警自身指标，再按场景补充服务、K8s、主机或依赖指标。
3. 查询计划分阶段执行，第一阶段少而准，第二阶段只在证据指向明确时扩展。
4. 模型只读取指标摘要，不直接读取完整时间序列。
5. 所有指标查询都要经过 RBAC、数据源范围、时间窗口、TopK 和超时控制。
6. 证据不足要明确输出缺口，不能把无数据解释成确定根因。

## 查询链路

```text
告警上下文
  -> 解析环境、服务、集群、命名空间、资源、metric_name、labels、触发时间
  -> 选择知识环境绑定的指标数据源
  -> 生成第一批指标查询计划
  -> 按预算执行 PromQL
  -> 压缩为趋势、基线、当前值、异常状态
  -> 必要时按异常方向扩展第二批
  -> 返回指标证据包
  -> 模型基于证据解释根因
```

## 查询预算

默认预算：

- 单次告警根因分析最多 8 条 PromQL。
- 默认查询告警前后或最近 60 分钟窗口。
- 默认步长 60 秒。
- 每条查询最多保留 5 条序列摘要。
- 完整时序点不直接交给模型，只保留当前值、基线、最大值、趋势和异常状态。

预算命中时，后端应停止扩展，并在证据包里说明“已达到查询预算”。

## 指标模板

第一阶段必查：

- 告警自带 `metric_name` 和 labels 生成的触发指标。
- 服务类告警的 RED 指标：请求量、错误率、P95 延迟。
- K8s 类告警的运行时指标：重启次数、CPU、内存、副本可用性。

第二阶段扩展：

- 错误率和延迟同时异常时，扩展依赖、入口和下游组件指标。
- CPU 或内存异常时，扩展节点、容器限制和 throttling。
- Pod 重启异常时，扩展 OOMKilled、CrashLoopBackOff、镜像拉取和事件证据。
- 请求量突降或突增时，扩展入口流量、网关和限流指标。

## 证据包结构

```json
{
  "summary": {
    "alert_id": 123,
    "planned_count": 6,
    "executed_count": 6,
    "abnormal_count": 2,
    "missing_count": 1,
    "budget": 8,
    "window": {
      "start": "2026-06-17T10:00:00+08:00",
      "end": "2026-06-17T11:00:00+08:00"
    }
  },
  "evidence": [
    {
      "name": "服务 5xx 错误率",
      "status": "abnormal",
      "trend": "up",
      "latest": 4.8,
      "baseline": 0.2,
      "evidence_weight": "strong"
    }
  ]
}
```

## 模型使用约束

模型回答时只能基于证据包说明：

- 哪些指标异常。
- 异常是否接近告警时间窗口。
- 哪些指标正常，能否作为反证。
- 指标证据是否支持日志、Trace、K8s 或变更证据指向的根因。
- 哪些关键指标缺失，需要继续补查。

模型不能：

- 编造未查询的指标。
- 把“无数据”解释成“正常”。
- 直接输出未经过平台后端执行的 PromQL 作为事实。
- 忽略数据源环境和告警环境不一致的风险。

## 当前落地范围

AIOps 2.1 已落地：

- 新增 `query_alert_metrics` 只读平台工具。
- 告警根因分析自动调用指标证据包，而不是只查单条 PromQL。
- 支持告警原始指标、服务 RED 和 K8s 运行态指标模板。
- 复用 `ops.metric.query` 权限。
- 复用知识环境的 `metric_datasource_ids` 选择指标数据源。
- 返回指标计划、摘要、异常项和证据不足项。

可按实际数据源继续扩展：

- 数据库、Redis、Kafka、网关等常见运行组件模板。
- 基于异常方向的第二阶段扩展。
- 与系统态势拓扑结合的上下游指标扩展。
- 指标模板后台配置和审计页面。
