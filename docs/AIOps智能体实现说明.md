# AIOps 智能体实现说明

本文记录 Xing-Cloud AIOps 智能体的当前实现方式、关键模块、运行链路与工程边界，便于理解平台如何把模型能力接入真实运维控制面。

## 1. 总体目标

AIOps 智能体不是脱离平台的数据问答机器人，而是把可观测性、事件中心、任务中心、工单系统、容器管理和资源上下文统一抽象为 LLM 可调用的工具，再由模型结合工具事实给出结论、证据和待确认动作。

核心原则：

- 只通过平台内数据和受控工具回答。
- 优先使用 LLM tool-calling 选择工具，不依赖前端关键词拼装查询。
- 查询类问题直接返回事实与解释，动作类问题必须先生成草稿并等待确认。
- 回答必须基于工具事实，证据不足时明确说明不足。
- 会话、工具调用、待确认动作和执行结果都要可审计。

## 2. 整体架构

当前实现分为 6 层：

1. 前端聊天层：`frontend/src/components/aiops/AIOpsChatWidget.vue`
2. AIOps API 层：`backend/aiops/views.py`
3. 智能体调度层：`backend/aiops/services.py`
4. 平台内置 MCP 工具层：统一注册可观测性、事件、任务、工单、容器和资源上下文等工具。
5. 外部 MCP 接入层：通过 `AIOpsMCPServer` 配置外部 HTTP / stdio MCP。
6. Skill 与回答整形层：用领域模板约束输出结构和安全边界。

## 3. 问答链路

以“分析生产 workorder-center 最近异常”为例：

1. 前端发送问题到同步或异步消息接口。
2. 后端创建用户消息和占位助手消息。
3. 调度层加载模型配置、启用的 MCP、启用的 Skill、历史消息和页面上下文。
4. 第一阶段 LLM 做 tool-calling 规划。
5. 后端按工具白名单执行 MCP / 平台工具。
6. 工具返回结构化事实，形成事实集。
7. 后端生成一版基于事实的兜底草稿。
8. 第二阶段 LLM 结合事实集和 Skill 模板做回答整形。
9. 若整形结果不合格，回退到代码兜底草稿。
10. 返回最终消息、进度事件、工具调用、证据块和待确认动作。

这里的“进度事件”是平台定义的可控事件，例如加载 MCP、模型规划、工具执行、生成回复、Skill 整形；不是模型原始推理全文。

## 4. MCP 与 Skill 的职责

MCP 负责把平台能力标准化为模型可调用的工具。当前内置方向包括：

- 可观测性 MCP
- 事件中心 MCP
- 任务中心 MCP
- 工单系统 MCP
- 容器管理 MCP
- 资源上下文 MCP

Skill 不负责查数据，主要负责两件事：

- 约束模型如何组织回答。
- 让同类问题的输出结构稳定。

典型约束包括：

- 告警 / 异常分析：输出“结论、依据、建议操作”。
- 工单汇总：输出“范围、数量、明细、建议”。
- 任务生成：输出“目标、动作、风险、确认提示”。

## 5. 工具执行边界

模型只负责提出“要调用哪个工具、参数是什么”，真正执行永远在后端完成。

后端工具层负责：

- RBAC 权限校验。
- 参数清洗与资源范围约束。
- 超时、限流和异常兜底。
- 统一审计工具调用。
- 对接内置工具和外部 MCP。

因此，智能体可以理解自然语言，但不能绕过平台 API、权限、审计和确认流程直接操作资源。

## 6. 双阶段回答

单阶段 tool-calling 常见问题是工具已经查到事实，但最终回答过短、格式不稳定或漏掉建议动作。当前实现采用双阶段：

- **第一阶段取证**：LLM 选择工具，平台执行工具并返回事实。
- **第二阶段整形**：LLM 基于事实和 Skill 模板生成最终回答。
- **代码兜底**：当二阶段输出为空、过泛或异常时，回退到后端草稿。

这套机制的目标是兼顾自然表达、事实约束和可用性。

## 7. 待确认动作

任务生成、自愈、执行类操作都不会直接落地。平台会先生成待确认动作，用户确认后再进入任务中心或相应执行流程。

这样可以避免：

- 用户只是咨询，系统却直接创建任务。
- 目标主机、集群或命名空间识别不清。
- 高风险指令未经确认执行。

## 8. 前端体验层

关键组件：

- `frontend/src/components/aiops/AIOpsChatWidget.vue`

当前前端能力包括：

- 会话历史。
- 异步轮询与流式显示。
- 进度事件折叠。
- 待确认动作卡片。
- 工具调用与证据展示。
- 页面上下文带入。

顶部导航、右下角浮窗和 AIOps 页面入口共用 `xing-cloud-aiops-open` 事件，确保不同入口打开同一个智能助手面板。

## 9. 审计与权限

智能体配置页中的审计能力覆盖：

- 会话审计。
- 工具调用审计。
- 模型调用与成本审计。
- 待确认动作审计。
- 协同任务与 Runbook 审计。

删除会话、调用工具、生成动作和执行动作均受 RBAC 控制。前端隐藏入口只是体验收敛，后端权限校验是安全边界。

## 10. 工程边界

当前实现的边界如下：

- Skill 整形是主路径，代码兜底是最后保障。
- 外部 MCP 通过配置接入，实际可用性依赖外部服务健康、鉴权和网络环境。
- 指标、日志、Trace、事件等分析质量依赖数据源覆盖范围和字段质量。
- 进度事件用于解释平台执行链路，不暴露原始 Chain-of-Thought。
- 动作类能力必须经过预检、确认、权限校验和审计记录。

## 11. 一句话总结

Xing-Cloud AIOps 智能体的本质是：

**用 LLM 做理解和规划，用 MCP 提供事实能力，用 Skill 约束回答结构，用平台后端负责安全、审计、执行与兜底。**
