# Xing-Cloud 用户使用文档

本文面向日常使用 Xing-Cloud 的运维、SRE、研发和平台管理员。

## 使用前准备

1. 使用具备对应权限的账号登录平台。
2. 先维护资产登记、平台管理、日志数据源、指标数据源和告警规则。
3. 使用 AIOps 前，建议先补齐知识图谱环境、事件环境、任务资源和数据源关系。
4. 所有执行类动作都应确认环境、系统、资源范围和风险后再提交。

## 运行概览

运行概览是“智能运维平台驾驶舱”，主要给管理者查看：

- 本月 SLA 状态：达标、风险、未达标。
- 年度 99.96% 目标是否可达成。
- 数据库、中间件、容器平台、网络、服务器等产品 SLA。
- 本月工单数量、处理及时率、超时和失败情况。
- 告警数量、级别分布、未确认告警和风险项。

SLA 当前按灾难级别告警的持续时长计算。

## 资产登记

资产登记用于维护平台要管理的资产基础信息：

- 一级业务。
- 环境。
- 主机或 K8S 资源。
- 运维负责人。
- 项目负责人。
- 说明和扩展信息。

建议先维护环境和业务目录，再登记具体资产。

## 平台管理

平台管理包含 K8S 集群和容器环境：

- 新增 K8S 集群时，平台会给出只读用户或管理用户的 kubeconfig 创建脚本。
- 脚本只负责创建 ServiceAccount、绑定权限并输出 kubeconfig。
- 容器环境用于登记 Docker 主机，便于查看容器、镜像、日志和执行容器操作。

## 可观测性

当前可观测性包含：

- 平台总览。
- 监控看板。
- 日志中心。
- 告警中心。

### 平台总览

平台总览用于集中维护可观测性数据源状态：

- 日志数据源：Loki、ELK/Elasticsearch、ClickHouse。
- 指标数据源：Prometheus 兼容接口。
- 告警规则：平台主动评估的规则。

### 监控看板

监控看板使用平台原生设计，不依赖外部看板跳转或嵌入。当前只保留三类看板：

- 服务器看板。
- K8S 集群看板。
- 日志看板。

默认看板会优先使用默认指标源和默认日志源展示。

### 日志中心

日志中心支持 ELK、Loki、ClickHouse 日志检索。

ClickHouse 可以在一个数据源下配置多个日志集合，例如：

- K8S 容器日志。
- K8S 集群事件。
- Ingress 访问日志。

日志中的 `trace_id`、`span_id`、`request_id` 可以作为普通字段检索，但平台当前不提供链路追踪页面。

### 告警中心

告警中心以平台自带告警能力为主：

- 平台基于 Prometheus 指标、ClickHouse 日志、K8S 资源/事件、SLA 或内置规则主动触发告警。
- 告警可进入聚合、抑制、静默、升级和通知流程。
- AI 研判会根据证据生成结论、风险和建议。
- 通知规则可主动推送到邮件、短信、语音、钉钉、飞书、企微等渠道。

客户系统不需要向告警中心配置 Webhook 回调地址。外部事件应进入事件中心。

## 事件中心

事件中心用于沉淀发布、变更、任务、外部系统事件和平台关键操作，按时间线还原排障上下文。

常用操作：

- 按环境、系统、服务、事件源、结果和时间范围过滤事件。
- 查看故障窗口内的变更、发布、任务和外部事件。
- 维护事件环境和事件源。

事件中心的外部 Webhook 是事件审计能力，不属于告警中心反向接入。

## 任务中心

任务中心用于维护执行资源、下发运维任务、复用任务模板和查看执行结果。

支持：

- SSH 连通性检查。
- 主机指标刷新。
- 批量命令执行。
- Ansible Playbook 执行。
- 服务状态巡检。
- K8S Pod 重启。
- K8S Pod 命令执行。
- K8S 工作负载伸缩。

## 工单系统

工单系统包含应用发布、审批流、SQL 审计和事务工单。生产动作建议通过工单或任务中心形成可审计闭环。

## AIOps

AIOps 用于通过自然语言完成运维问答、日志分析、告警研判、证据汇总、处置建议和任务草稿生成。

使用建议：

- 提问时带上环境、系统、服务、时间范围和现象。
- 排障类问题优先说明开始时间、影响服务、看到的告警或错误。
- 需要执行命令或生成任务时，先让 AIOps 生成建议或草稿，再由人工确认。
- 高风险动作必须由平台权限、预检和人工确认把关。

## 注意事项

- 当前测试阶段不保留历史演示数据，部署时可以使用全新的数据库和 Redis。
- 数据源、事件环境、任务资源和知识图谱环境应保持同一套环境、系统、服务命名规范。
- 新增系统或服务后，应同步补齐可观测性、事件中心、任务资源和知识图谱配置。
